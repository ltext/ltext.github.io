λtext

General-Purpose Templating Utility

Reduce Duplication

The Copy-paste command is an essential for anyone using a computer. However, every time you use it, you need to manually duplicate the data. With λtext, you can parameterize files themselves.

Built with Precision

With Lambda Calculus as its underlying engine, λtext features both generality and precision - it's Hindley-Milner type system gives us the best of both worlds; polymorphism and correctness. Function application is now our copy-and-paste command.

Open Source

λtext is BSD Licenced - you are free to do what you want with the project. It's written in the advanced programming language Haskell, with its code available on github. Total lines of code? 978.

How It Works

Step 1: Have a Text File

The first thing you need is a text file you want to parameterize. This can be Html, JavaScript, Haskell, Bash, TeX, anything. In this example, we'll use a plaintext file foo: 

Woo hoo! I'm a Text File!

Here's some content that I /really/
don't want inside here. Oh well!

Step 2: Create a Parameter

Just like lambda calculus, we create arity via abstraction - the act of creating a parameter. In λtext it's very easy - all we need to do is, on the first line, declare our parameters inside delimiters. Then, we use those declared variables by referencing their names with the same delimiters. We are free to choose the syntax we wish - Jade-style delimiters with {{ }}, Html-friendly delimiters via comments <!-- -->, etc. Here's our new and improved foo: 

$$ x %%
Woo hoo! I'm a Text File!

$$ x %%

Step 3: Have Another File to Substitute

Now that we have our function/file foo, we need some content to use as it's input. Easy enough, we can just plug in the content we ripped-out into the file bar: 


Here's some content that I /really/
don't want inside here. Oh well!

Note the empty first line - that way ltext doesn't think some of our words are actually delimiters.

Step 4: Apply the Files/Functions with λtext

Lastly, to use our functions, we must apply one to another - the other counterpart to lambda calculus. To do this, invoke the ltext command from a terminal: 

$ ltext "foo bar"

Woo hoo! I'm a Text File!

Here's some content that I /really/
don't want inside here. Oh well!

Normally, the result of the expression we supply to ltext (ie: the files we wish to apply) just pumps out stdout. But, if that's unwanted, there's a flag for that!

Advanced Usage

There are a few other things to consider when working with λtext - type checking, partial application, and debugging! *gasp*

Type Checking

To inspect the type of a file or an expression, simply include the -t option during invocation: 

$ ltext "foo" -t

foo :: Text -> Text

You can read this as "foo takes in Text and returns Text", or "foo is a function from Text to Text". This is useful if you are getting type matching errors, or want to get a description of a file. We can also check the type signature of expressions directly: 

$ ltext "\x -> x" -t

λx. x :: ∀ a0. a0 -> a0

This is a little technical, but we are creating a polymorphic function \x -> x, which has a type from for all a0 to a0 (you may see these cryptic type variables instead of Text from time to time). This would happen if, for instance, you made an "apply"-like template: 

%% f x %%

asdf

%% f x %%

If you check the type of apply, you can see it's... 

$ ltext -t "apply"

apply :: ∀ a1. (a1 -> Text) -> a1 -> Text

Meaning that x can be content of any type, so long as it's the same input as f.

Partial Application

Just like lambda calculus, we can partially apply templates to each other. Take our apply template from before - that's a good example to work though. If we apply our foo file from before to apply, we should get a template from Text -> Text - the parameter to foo being unified with the x parameter in apply: 

$ ltext -t "apply foo"

apply foo :: Text -> Text

It turns out, we can actually render this partially applied template, but only if we explicitly supply new delimiters for the result: 

$ ltext "apply foo" --left "{{" --right "}}"

{{ x }}

asdf

Woo hoo! I'm a Text File!

{{ x }}

Debugging

If you toyed around with the apply function from earlier, you may have seen a daunting error: 

$ ltext "\f -> apply f bar"

ltext: Error: Result has literals in sub expression - `λf. (#apply) <> (f #bar)` - cannot render soundly.

This is intended! This makes sure that the templates we render are actually renderable.

In laymen's terms, a template can only be rendered if all Text data that is concatenated (<>) is at the "top level" - if after reducing our expression, we wind up applying text to an unknown function, or applying unknown input to a file, we can't fully substitute the text, and are left with dangling references to files.

We can check the type of these expressions, but we can't render them. To help us see what is going on under-the-hood, it's sometimes helpful to "show" the underlying expression: 

$ ltext "\f -> apply f bar" --show

λf. (#apply) <> (f #bar) :: (Text -> Text) -> Text

By looking at this type, you can see that we still need a continuation-passing-style function f before we can get our result. Looking at the expression itself, you can see that we're applying the bar file to the unknown function f - therefore we can't substitute it in our result.

Note: We prepend file names with # in an attempt to distinguish them from normal terms.

So, if you ever get this error, you know that there's something you still need to apply before you can fully render your expression. I hope that clears up some headaches!
